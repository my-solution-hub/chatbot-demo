# Requirements Document

## Introduction

This feature delivers a minimal starter demo showcasing Amazon Nova Sonic v2, an end-to-end speech-to-speech foundation model, for customer-facing demonstrations. The demo captures spoken user input, streams it to the Nova Sonic model, and plays back the model's spoken response. To illustrate tool use, the demo exposes a small set of simple API tools (for example, a current time tool and a weather lookup tool) that Nova Sonic can invoke through tool calls when the spoken user input requires them. The demo is intentionally minimal: few dependencies, clear code structure, and easy to run locally so that customers can understand the core interaction pattern before additional features are layered on.

## Glossary

- **Nova_Sonic_Demo**: The complete local application being specified, including audio capture, Nova Sonic session management, tool execution, and audio playback.
- **Audio_Capturer**: The component responsible for capturing microphone input from the user and forwarding PCM audio frames to the Nova Sonic session.
- **Audio_Player**: The component responsible for playing back PCM audio output received from the Nova Sonic session through the local speakers.
- **Sonic_Session**: The bidirectional streaming session with the Amazon Bedrock Nova Sonic v2 model (model identifier `amazon.nova-sonic-v2:0`), used for end-to-end speech understanding and speech generation.
- **Tool_Registry**: The in-process collection of tool definitions and handler functions that the Nova_Sonic_Demo exposes to the Sonic_Session at session start.
- **Tool_Handler**: A local Python function that implements a single tool and returns a JSON-serializable result.
- **Demo_CLI**: The command-line entry point that starts the demo, initializes the Sonic_Session, and manages the demo lifecycle until the user exits.
- **Tool_Call**: A structured request emitted by the Nova Sonic model during a session instructing the Nova_Sonic_Demo to execute a named tool with a given set of arguments.
- **Tool_Result**: The JSON-serializable response returned from a Tool_Handler and sent back to the Sonic_Session in reply to a Tool_Call.

## Requirements

### Requirement 1: End-to-End Speech Interaction

**User Story:** As a customer evaluating Nova Sonic, I want to speak to the demo and hear a spoken response, so that I can experience end-to-end speech-to-speech interaction without typing.

#### Acceptance Criteria

1. WHEN the Demo_CLI is started, THE Nova_Sonic_Demo SHALL open a bidirectional streaming Sonic_Session with the `amazon.nova-sonic-v2:0` model on Amazon Bedrock within 10 seconds of startup.
2. WHILE a Sonic_Session is active, THE Audio_Capturer SHALL stream microphone audio to the Sonic_Session as 16-bit PCM at 16 kHz mono.
3. WHILE a Sonic_Session is active, THE Audio_Player SHALL play audio output received from the Sonic_Session as 16-bit PCM at 24 kHz mono.
4. WHEN the Sonic_Session emits an end-of-utterance event indicating the user has stopped speaking, THE Sonic_Session SHALL begin producing a spoken response generated by the Nova Sonic v2 model within 5 seconds, regardless of whether the utterance triggered a Tool_Call.
5. WHEN the user presses Ctrl+C in the Demo_CLI, THE Nova_Sonic_Demo SHALL stop the Audio_Capturer, stop the Audio_Player, close the Sonic_Session, release all audio device handles, and exit the process with status code 0 within 5 seconds.
6. IF the Sonic_Session cannot be established within 10 seconds due to authentication failure, network failure, unsupported region, or model unavailability, THEN THE Nova_Sonic_Demo SHALL write a human-readable error message to stderr that identifies the failure category and includes the underlying error, and SHALL exit with a non-zero status code.
7. IF no microphone input device or no speaker output device is available when the Nova_Sonic_Demo starts, THEN THE Nova_Sonic_Demo SHALL write a human-readable error message to stderr identifying the missing device and SHALL exit with a non-zero status code without opening a Sonic_Session.

### Requirement 2: Tool Registry Exposed to the Model

**User Story:** As a customer, I want the demo to expose a small set of simple tools to Nova Sonic, so that I can see how the model decides to call tools based on spoken input.

#### Acceptance Criteria

1. WHEN a Sonic_Session is initialized, THE Nova_Sonic_Demo SHALL register exactly two tools with the Sonic_Session: `get_current_time` and `get_weather`.
2. THE Tool_Registry SHALL provide `get_current_time` as a tool that takes an optional `timezone` string argument (IANA timezone name, defaulting to "UTC" when omitted) and returns a JSON object containing an ISO 8601 timestamp and a resolved timezone field that echoes the resolved IANA timezone name.
3. THE Tool_Registry SHALL provide `get_weather` as a tool that takes a required `city` string argument (length 1 to 100 characters and non-empty after trimming whitespace) and returns a JSON object containing the city name, a textual condition from the predefined set {"sunny", "cloudy", "rainy", "snowy", "windy"}, and an integer temperature in Celsius in the range -50 to 50 inclusive.
4. THE Tool_Registry SHALL supply a JSON Schema for each tool's input arguments at session initialization, including the tool name, a description of 1 to 200 characters, and typed parameters where each parameter declares a `type` and required properties are explicitly listed in the schema's required array.
5. WHERE no live weather source is configured, WHEN the `get_weather` tool is invoked, THE Tool_Handler SHALL return a mocked response such that, given identical input arguments within a single process run, the returned output is identical, so that the demo runs without external API credentials.
6. IF a Tool_Call references a tool name not present in the Tool_Registry, THEN THE Nova_Sonic_Demo SHALL return a Tool_Result containing an `error` field with the value `unknown_tool` and the offending tool name.
7. IF `get_current_time` is invoked with an invalid or unknown IANA timezone value, THEN THE Tool_Handler SHALL return a Tool_Result containing an `error` field with the value `invalid_timezone`.
8. IF `get_weather` is invoked with a missing `city` argument or a `city` argument that is empty after trimming whitespace, THEN THE Tool_Handler SHALL return a Tool_Result containing an `error` field with the value `invalid_arguments`.

### Requirement 3: Tool Call Triggering from Spoken Input

**User Story:** As a customer, I want to ask the demo questions like "what time is it?" or "what is the weather in Seattle?" by voice, so that I can observe Nova Sonic triggering the correct tool based on what I said.

#### Acceptance Criteria

1. WHEN the Sonic_Session emits a Tool_Call for a registered tool, THE Nova_Sonic_Demo SHALL dispatch the call to the matching Tool_Handler using the arguments supplied by the model within 500 milliseconds of receiving the Tool_Call.
2. WHEN a Tool_Handler returns a Tool_Result, THE Nova_Sonic_Demo SHALL send the Tool_Result back to the Sonic_Session associated with the originating Tool_Call identifier within 500 milliseconds of the Tool_Handler returning.
3. WHEN a Tool_Result is delivered to the Sonic_Session, THE Sonic_Session SHALL produce a spoken response audio stream generated by the Nova Sonic v2 model for the same user turn, incorporating the Tool_Result content, without requiring additional user speech.
4. IF a Tool_Handler raises an exception, THEN THE Nova_Sonic_Demo SHALL catch the exception, return a Tool_Result containing an `error` field with a human-readable message of at most 200 characters, release any resources acquired by the failing Tool_Handler, and keep the Sonic_Session open and accepting further audio input.
5. WHEN a Tool_Handler exception has been handled, THE Nova_Sonic_Demo SHALL reset its per-call tool execution state such that a subsequent Tool_Call is dispatched using only that call's arguments, the Sonic_Session remains open, and microphone audio continues to be accepted.
6. WHEN a Tool_Call is dispatched, THE Nova_Sonic_Demo SHALL write a single-line JSON log entry containing the tool name and arguments to standard output before the Tool_Handler is executed.
7. IF a Tool_Handler has not returned a Tool_Result within 10 seconds of dispatch, THEN THE Nova_Sonic_Demo SHALL return a Tool_Result with the `error` field set to `tool_timeout`, cancel the Tool_Handler, release any resources acquired by the cancelled Tool_Handler, and keep the Sonic_Session open and accepting further audio input.
8. IF a Tool_Call's arguments fail JSON Schema validation against the registered schema for that tool, THEN THE Nova_Sonic_Demo SHALL return a Tool_Result with the `error` field set to `invalid_arguments` without invoking the Tool_Handler, and keep the Sonic_Session open and accepting further audio input.

### Requirement 4: Local Runability and Minimal Dependencies

**User Story:** As a developer preparing the demo, I want to run the project locally with a small number of dependencies and a single command, so that I can show the demo on any laptop with minimal setup.

#### Acceptance Criteria

1. THE Nova_Sonic_Demo SHALL be implemented using Python version 3.10, 3.11, 3.12, or 3.13.
2. THE Nova_Sonic_Demo SHALL declare all third-party runtime dependencies with pinned versions in a single `requirements.txt` file located at the project root.
3. THE Nova_Sonic_Demo SHALL limit third-party runtime dependencies listed in `requirements.txt` to a maximum of five packages, covering AWS SDK access, audio capture and playback, and asynchronous streaming.
4. WHEN the user runs the command `python -m nova_sonic_demo` from the project root with the declared dependencies installed and AWS credentials resolvable, THE Demo_CLI SHALL start the demo and begin a Sonic_Session within 10 seconds.
5. THE Nova_Sonic_Demo SHALL read AWS credentials and the AWS region from the standard AWS SDK credential chain (environment variables, shared config file, or IAM role) without requiring any project-specific configuration file.
6. WHERE the environment variable `AWS_REGION` is set to a Bedrock region that supports Nova Sonic v2, THE Nova_Sonic_Demo SHALL use that region for the Sonic_Session.
7. IF AWS credentials cannot be resolved from the standard AWS SDK credential chain when the Demo_CLI starts, THEN THE Demo_CLI SHALL terminate with a non-zero exit code and display an error message indicating that AWS credentials are missing or invalid.
8. IF the resolved AWS region does not support Nova Sonic v2, THEN THE Demo_CLI SHALL terminate with a non-zero exit code and display an error message indicating that the configured region is unsupported.
9. THE project root SHALL contain a `README.md` file that documents the supported Python versions, dependency installation steps, the run command `python -m nova_sonic_demo`, AWS credential and region setup instructions, and at least one example spoken prompt for every tool supported by the demo.

### Requirement 5: Observability for Demo Audiences

**User Story:** As a presenter running the demo for customers, I want visible feedback in the terminal about what the model is doing, so that the audience can follow along with the speech interaction.

#### Acceptance Criteria

1. WHEN the Sonic_Session starts, THE Demo_CLI SHALL print a startup banner line to standard output that includes the model identifier and the active AWS region, terminated by a single newline character.
2. WHILE a Sonic_Session is not yet active, THE Demo_CLI SHALL discard transcript, Tool_Call, and Tool_Result output at emission time without retaining them in memory and without deferring them for later flush once the Sonic_Session becomes active.
3. WHEN the Audio_Capturer begins streaming microphone audio after the Sonic_Session has started, THE Demo_CLI SHALL print a single line to standard output beginning with the literal prefix `LISTENING:` followed by a single space and a human-readable phrase indicating that the demo is listening, terminated by a single newline character.
4. WHEN the Sonic_Session receives a final transcript segment of the user's speech, THE Demo_CLI SHALL print a single line to standard output beginning with the literal prefix `USER:` followed by a single space and the transcript text, terminated by a single newline character, and SHALL not print partial or interim transcript segments.
5. WHEN the Sonic_Session produces a final transcript segment of the model's spoken response, THE Demo_CLI SHALL print a single line to standard output beginning with the literal prefix `ASSISTANT:` followed by a single space and the transcript text, terminated by a single newline character, and SHALL not print partial or interim transcript segments.
6. WHEN a Tool_Call is received during an active Sonic_Session, THE Demo_CLI SHALL print a single line to standard output beginning with the literal prefix `TOOL_CALL:` followed by a single space, the tool name, a single space, and the tool arguments serialized as a single-line JSON value, terminated by a single newline character.
7. WHEN a Tool_Result is returned to an active Sonic_Session, THE Demo_CLI SHALL print a single line to standard output beginning with the literal prefix `TOOL_RESULT:` followed by a single space, the tool name, a single space, and the tool result serialized as a single-line JSON value, terminated by a single newline character.
8. IF a Tool_Call's arguments or a Tool_Result's body cannot be JSON-serialized, THEN THE Demo_CLI SHALL print a fallback line to standard output using the same `TOOL_CALL:` or `TOOL_RESULT:` prefix and tool name, replacing the non-serializable payload with the literal string `<non-serializable>`, terminated by a single newline character, and the Sonic_Session SHALL continue unaffected.
