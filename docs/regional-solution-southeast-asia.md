# 东南亚区域语音方案选型

## 背景

Nova Sonic 是端到端的语音对话模型（Speech-to-Speech），目前主要支持英语及部分欧洲语言。对于东南亚市场（泰语、越南语、印尼语、马来语、菲律宾语、高棉语、缅甸语、老挝语），需要评估替代方案。

## 方案对比：Nova Sonic vs ASR + LLM + TTS 管道

### 延迟对比

| 指标 | Nova Sonic (Speech-to-Speech) | ASR + LLM + TTS 管道 |
|------|-------------------------------|---------------------|
| 首字节音频时间 (TTFA) | **~1.1 秒** | **~2-4 秒**（累加） |
| ASR 延迟 | 不适用（内置） | 270-520 ms |
| LLM 推理延迟 | 不适用（内置） | 500-1500 ms |
| TTS 延迟 | 不适用（内置） | 200-600 ms |
| 网络跳数 | 1 次（单一双向流） | 3-4 次（每个组件一次） |
| 打断支持 | 原生支持，无额外延迟 | 需要额外 VAD + 编排逻辑 |

**结论：Nova Sonic 在延迟上有明显优势（约 2-3 倍更快）。** 端到端模型避免了级联架构中每个阶段累加的延迟和错误传播。但管道方案在语言覆盖和组件可定制性上更灵活。

### 架构复杂度对比

| 维度 | Nova Sonic | ASR + LLM + TTS |
|------|-----------|-----------------|
| 组件数量 | 1 个模型 | 3-4 个独立服务 |
| 开发复杂度 | 低（单一 API） | 高（编排、错误处理、流式对接） |
| 运维成本 | 低 | 高（每个组件独立监控/扩缩） |
| 可定制性 | 有限（不能替换子组件） | 高（每个环节可独立调优/替换） |
| 语言扩展 | 受限于模型支持 | 灵活（选择最佳 ASR/TTS 组合） |

### 何时选择哪种方案

**选 Nova Sonic：**
- 目标语言在 Sonic 支持范围内
- 延迟是核心指标（客服、实时助手）
- 希望简化架构和运维

**选 ASR + LLM + TTS 管道：**
- 需要支持 Nova Sonic 不覆盖的语言（如东南亚语言）
- 需要对单个组件精细调优（如领域特定 ASR）
- 需要跨语言对话或实时翻译

## 东南亚语言 ASR + TTS 厂商覆盖

### Nova Sonic 实测结果（2026-05-18）

使用本项目 Web UI 对 Nova Sonic 进行东南亚语言实测，system prompt 为默认英文助手：

| 语言 | Sonic 能否识别 | 能否正确回复 | 备注 |
|------|-------------|------------|------|
| 印尼语 🇮🇩 | ✅ | ✅ 用印尼语回复 | ASR 有小偏差（"apa"→"abang"），但语义理解正确 |
| 马来语 🇲🇾 | ✅ | ✅ 用马来/印尼语回复 | 和印尼语表现一致（两者高度互通） |
| 泰语 🇹🇭 | ❌ | — | 完全无法识别 |
| 越南语 🇻🇳 | ❌ | — | 被当成中文处理（"xin chào"→"先找"/"新巢"） |

**规律：拉丁字母 + 非声调语言**（印尼/马来）Sonic 能泛化处理；**声调语言**（泰/越）需要专门训练，当前完全不支持。

> 注：印尼语和马来语虽未列入 Sonic 官方支持语言，但实测可用。这意味着对于印尼/马来市场，可以直接使用 Sonic 而无需管道方案，但需要接受 ASR 精度不如官方支持语言的事实。

### 关键发现

1. **AWS 的短板**：Transcribe 在 2024 年 10 月扩展了流式支持（含越南语、印尼语、马来语、Tagalog），ASR 覆盖不错。但 **Polly 完全没有东南亚语言的 TTS**，这是最大的缺口。

2. **ElevenLabs 是强力补充**：70+ 语言的 TTS，覆盖主要东南亚语言（泰/越/印尼/马来/菲律宾），ASR 也有（Scribe）。但高棉/缅甸/老挝不支持。

3. **OpenAI Whisper**：ASR 覆盖最广（100+ 语言），但没有 TTS 产品。适合作为 ASR 层。

4. **没有一家能完美覆盖所有东南亚语言的 ASR + TTS**。高棉语和老挝语的 TTS 是行业空白。

## 推荐方案

### 方案 A：AWS 生态 + ElevenLabs TTS（推荐）

```
用户语音 → AWS Transcribe (ASR, 流式)
         → Amazon Bedrock LLM (理解 + 生成)
         → ElevenLabs TTS (语音合成)
         → 用户
```

- **优点**：ASR 和 LLM 留在 AWS，数据合规简单；ElevenLabs TTS 质量高
- **缺点**：TTS 需要外部服务，增加一跳延迟
- **覆盖**：泰/越/印尼/马来/菲律宾 ✅ | 高棉/缅甸/老挝 ❌

### 方案 B：混合路由（按语言选路径）

```
语言检测 → 英语/欧洲语言 → Nova Sonic（低延迟，最佳体验）
         → 东南亚语言   → ASR + Bedrock LLM + ElevenLabs TTS（管道方案）
```

- **优点**：支持语言用最优路径，不支持的语言有降级方案
- **缺点**：架构复杂，两套路径需要维护
- **适用**：多语言产品，核心市场是英语但需要覆盖东南亚

### 方案 C：Whisper (自部署) + Bedrock LLM + ElevenLabs TTS

```
用户语音 → Whisper (自部署在 EKS/SageMaker, ASR)
         → Amazon Bedrock LLM
         → ElevenLabs TTS API
         → 用户
```

- **优点**：ASR 语言覆盖最广（100+），可自定义模型
- **缺点**：自部署 Whisper 需要 GPU 资源，运维成本高
- **覆盖**：几乎所有东南亚语言 ASR ✅ | TTS 取决于 ElevenLabs 覆盖

## 延迟预估（管道方案 vs Sonic）

| 场景 | Nova Sonic | 管道方案（优化后） | 管道方案（未优化） |
|------|-----------|------------------|------------------|
| 简单问答 | ~1.1s | ~2.0-2.5s | ~3-4s |
| 工具调用 | ~1.5-2s | ~3-5s | ~5-7s |
| 打断响应 | <200ms | ~500-1000ms | 不支持或很慢 |

**管道方案优化手段：**
- 流式 ASR（边听边转）
- LLM 流式输出 + TTS 流式合成（不等完整文本）
- TTS filler（"嗯..."、"让我看看..."）填充等待时间
- 使用 Pipecat / LiveKit 等异步编排框架

即使优化后，管道方案仍比 Sonic 慢约 1 秒。但对于东南亚语言，这是当前唯一可行的路径。

## 总结

| 目标市场 | 推荐方案 |
|---------|---------|
| 英语为主，偶尔东南亚 | 方案 B（混合路由） |
| 印尼/马来为主 | **直接用 Nova Sonic**（实测可用，延迟最低） |
| 泰/越为主 | 方案 A（AWS Transcribe + Bedrock + ElevenLabs TTS） |
| 需要覆盖高棉/缅甸/老挝 | 方案 C（Whisper 自部署）+ 接受 TTS 缺失的降级 |
| 延迟敏感 + 英语 | 直接用 Nova Sonic |
